## 1. الخرائط والموقع الجغرافي

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| دقة GPS في المناطق النائية والمبانí الكثيفة | ظهور حرفيين غير قريبين فعلياً | استخدام Google Maps Flutter مع fallback لـ OpenStreetMap (flutter_map + latlong2) |
| استهلاك البطارية مع التتبع المستمر | تجربة مستخدم سيئة | استخدام Geolocator مع تحديثات متباعدة (10 دقائق) وليس تتبعاً لحظياً |
| تكاليف Google Maps API مع نمو المستخدمين | تكاليف تشغيلية مرتفعة | التبديل إلى OpenStreetMap + MapTiler للمرحلة الثانية |
| عرض المسافة بين العميل والحرفي | دقة الحساب | حساب المسافة بـ Haversine formula على Backend، وليس على الجهاز |

## 2. إشعارات Firebase (FCM)

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| تأخير الإشعارات على أجهزة Huawei (بدون خدمات Google) | فوات طلبات للحرفيين | استخدام HMS Push (Huawei Mobile Services) بالتوازي مع FCM |
| عدم وصول الإشعارات في الخلفية (Background) | عدم استجابة الحرفي | استخدام flutter_local_notifications مع FCM data messages (وليس notification) |
| تكامل الإشعارات مع حالة التطبيق (foreground/background/killed) | تعقيد التنفيذ | إنشاء خدمة إشعارات موحدة (NotificationService) تدير الحالات الثلاث |

## 3. رفع الصور والوسائط

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| رفع صور متعددة (معرض الحرفي) | حجم بيانات كبير، زمن انتظار طويل | ضغط الصور محلياً بـ flutter_image_compress قبل الرفع |
| عرض الصور بجودة منخفضة في القوائم وأصلية في الملف | أداء بطيء في القوائم | استخدام CachedNetworkImage مع兩種 أحجام (thumbnail + full) |
| تخزين الصور — أين؟ | تكاليف وسرعة | استخدام Firebase Storage أو AWS S3 مع CDN |
| إلغاء الرفع عند انقطاع الاتصال | تجربة مستخدم محبطة | استخدام مكتبة retry مع pause/resume upload |

## 4. الأداء (Performance)

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| عرض قائمة طويلة من الحرفيين | بطء في التمرير (jank) | استخدام ListView.builder مع ItemExtent وتجنب rebuild غير ضروري |
| تحميل التطبيق الأولي بطيء | ارتفاع نسبة التخلّي (bounce rate) | تطبيق التحميل البطيء (Lazy Loading) واستخدام Splash Screen مع Pre-cache |
| حجم التطبيق كبير (APK/IPA) | صعوبة التحميل على أجهزة ضعيفة | تفعيل shrinking (ProGuard/R8)، حذف المكتبات غير المستخدمة، استخدام App Bundles |
| استخدامات متعددة للـ State Management | تناقض في إدارة الحالة | توحيد State Management (يفضل Bloc/Cubit أو Riverpod) عبر كل التطبيق |

## 5. التحديات اللغوية والثقافية (RTL)

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| دعم العربية (RTL) والفرنسية (LTR) | تشوه في واجهة المستخدم | استخدام Flutter's built-in RTL support مع MaterialApp(rtl: true) |
| الأرقام والأسعار بالعربية | عرض غير صحيح | توحيد تنسيق الأرقام عبر intl package (NumberFormat) |
| اتجاه الخريطة مع RTL | الخريطة قد تنعكس | استخدام Google Maps Widget الخاص مع تحديد اتجاه مستقل عن اللغة |
| محاذاة النصوص المختلطة (عربي + فرنسي + أرقام) | نصوص غير مقروءة | اختبار جميع الشاشات باللغتين، استخدام TextDirection.auto |

## 6. التكامل مع البوابات الخارجية

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| بوابة الدفع المغربية CMI | لا يوجد Flutter SDK رسمي | استخدام WebView داخل التطبيق (flutter_inappwebview) للدفع |
| فتح واتساب مباشر مع رقم الحرفي | تجربة مستخدم غير سلسة | استخدام url_launcher مع رابط whatsapp://send?phone=... |
| تكامل OAuth (Google/Facebook) غير مستقر | فشل تسجيل الدخول | استخدام firebase_auth كطبقة وسيطة بدلاً من التكامل المباشر |
| تحديث السعر والمسافة آنياً | زمن استجابة طويل | استخدام WebSockets (Socket.IO) بدلاً منHTTP polling |

## 7. الاشتراكات والمدفوعات داخل التطبيق

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| متطلبات Apple App Store (30% عمولة) | خفض هامش الربح | عدم تضمين الدفع داخل iOS — تحويل المستخدم لصفحة دفع خارجية |
| إدارة دورات الاشتراك (تجديد، إلغاء، استرداد) | تعقيد منطق الأعمال | استخدام backend لإدارة حالة الاشتراك وليس التطبيق فقط |
| التعامل مع إلغاء الاشتراك في منتصف الشهر | محاسبة غير عادلة | سياسة: الاشتراك غير مسترد، يستمر حتى نهاية الدورة |

## 8. التوافق مع الأجهزة (Device Fragmentation)

| التحدي | التأثير | الحل المقترح |
|--------|---------|-------------|
| أجهزة Android ضعيفة (2GB RAM فأقل) | تجربة مستخدم بطيئة | اختبار على أجهزة منخفضة المواصفات (Redmi, Sam A系列) |
| أجهزة iOS قديمة (iPhone 6/7) | التطبيق لا يعمل | تحديد minimum iOS 14+ (iPhone 6s وما فوق) |
| اختلاف أحجام الشاشات (notch, punch-hole, Dynamic Island) | تشوه شاشة التطبيق | استخدام SafeArea و MediaQuery للتعامل مع مختلف الأحجام |
| مشاكل أذونات Android 13+ (جديدة) | الموقع والإشعارات لا تعمل | اختبار الأذونات الجديدة (POST_NOTIFICATIONS, ACCESS_FINE_LOCATION) |

## ملخص التوصيات

1. **بدء Proof of Concept** للخرائط والإشعارات قبل بدء التطوير الرئيسي
2. **توحيد State Management** مبكراً (Riverpod أو Bloc)
3. **اختبار RTL** من اليوم الأول وليس في النهاية
4. **بناء طبقة API Service** مركزية للتعامل مع الأخطاء وإعادة المحاولة
5. **تخصيص موارد كافية** لاختبار الأجهزة الضعيفة (30% من وقت الـ QA)
